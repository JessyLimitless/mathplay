# 🤖 3-5. [Case Study] 에이전틱 워크플로우: 스스로 계획하고 도구를 사용하는 AI

> **단일 추론을 넘어, 계획 → 실행 → 검증의 루프를 자율적으로 수행하는 미래.**

---

## 대화를 넘어서

지금까지 이 책에서 다룬 LLM의 모든 메커니즘 — 자기회귀 생성, 어텐션, 프롬프팅, 메타인지 — 은 하나의 전제 위에 서 있었다. **인간이 질문하고, AI가 답한다.** 단발성 질문-응답 루프.

그러나 현실의 복잡한 과제는 단발성 질의로 해결되지 않는다.

"다음 분기 마케팅 전략을 수립하라"는 과제를 생각해보자. 이것은 하나의 프롬프트로 처리할 수 있는 문제가 아니다. 시장 데이터를 수집하고, 경쟁사를 분석하고, 예산 제약을 확인하고, 복수의 전략 대안을 생성하고, 각 대안의 ROI를 추정하고, 최종 전략을 선택하고, 실행 계획을 작성해야 한다. 각 단계가 이전 단계의 결과에 의존하며, 전체 과정은 **계획 → 실행 → 검증**의 반복적 루프로 구성된다.

**에이전트(Agent)**는 이 루프를 AI가 자율적으로 수행하는 시스템이다.

---

## 에이전트 아키텍처: Planner → Executor → Critic

대부분의 AI 에이전트는 세 가지 구성 요소로 이루어진다.

### Planner (계획자)

과제를 분석하고, 하위 목표(sub-goal)로 분해하며, 실행 순서를 결정한다. 3-3절의 CoT가 단일 추론 내에서 문제를 단계별로 분해했다면, Planner는 **다수의 독립적 추론 단계**를 사전에 설계한다.

### Executor (실행자)

각 하위 목표를 실행한다. 핵심은 **도구 사용(Tool Use)**이다. 텍스트 생성만으로 해결할 수 없는 하위 목표 — 웹 검색, 데이터베이스 쿼리, 코드 실행, API 호출 — 를 외부 도구를 호출하여 수행한다.

### Critic (비판자)

각 단계의 결과를 평가하고, 계획의 수정이 필요한지 판단한다. 3-3b절의 자기 수정 루프가 에이전트 수준으로 확장된 것이다.

---

## 도구 사용(Tool Use)의 수학

에이전트의 핵심 능력은 **언제, 어떤 도구를 사용할지를 결정하는 것**이다. 이것을 조건부 확률의 프레임으로 기술할 수 있다.

> ### P(action | state, goal)

- **state**: 현재 상태. 지금까지 수행된 단계의 결과, 수집된 정보, 남은 하위 목표.
- **goal**: 최종 목표.
- **action**: 다음에 수행할 행동. "웹 검색", "코드 실행", "사용자에게 질문", "텍스트 생성" 등.

이 구조는 강화학습(Reinforcement Learning)의 **정책(policy)** 함수와 동형이다. 에이전트는 매 순간 현재 상태와 목표를 참조하여, 목표 달성 확률을 최대화하는 행동을 선택한다. 제1장의 기차가 다음 토큰을 선택했듯이, 에이전트는 다음 행동을 선택한다. 단, 선택의 공간이 "어휘 사전의 토큰"에서 **"사용 가능한 도구와 행동"**으로 확장된 것이다.

---

## MCP와 Function Calling: 기술적 현주소

에이전트가 도구를 사용하려면, 도구와 소통하는 표준화된 인터페이스가 필요하다.

### Function Calling

모델이 "지금 이 도구를 이 파라미터로 호출해야 한다"고 판단하여, 구조화된 도구 호출 명령을 출력하는 기능이다. 예를 들어, 사용자가 "내일 서울 날씨를 알려줘"라고 하면, 모델은 텍스트 답변을 생성하는 대신 `weather_api(location="Seoul", date="tomorrow")`라는 함수 호출을 출력한다. 시스템이 이 호출을 실행하고, 결과를 모델에게 돌려주면, 모델은 그 결과를 바탕으로 자연어 답변을 생성한다.

### MCP (Model Context Protocol)

2024년 Anthropic이 제안한 프로토콜로, 다양한 외부 도구와 데이터 소스를 표준화된 방식으로 LLM에 연결하는 인터페이스다. 에이전트가 Google Drive, Slack, 데이터베이스, API 등 다양한 외부 서비스를 **일관된 프로토콜**로 접근할 수 있게 한다.

MCP의 의의는 기술적 표준화에 있다. 각 도구마다 별도의 연결 방식을 구현하는 대신, 하나의 프로토콜로 모든 도구를 연결할 수 있다면, 에이전트의 능력 확장이 극적으로 용이해진다.

---

## 🔮 전망: 단일 모델에서 다중 에이전트 협업으로

현재의 AI 에이전트는 대부분 **단일 LLM이 모든 역할**(계획, 실행, 비판)을 수행한다. 그러나 진화의 방향은 **다중 에이전트 시스템(Multi-Agent System)**을 가리킨다.

3-4절에서 우리는 발산적 과제와 수렴적 과제에 다른 모델이 최적이라는 것을 확인했다. 이 원리를 에이전트 아키텍처에 적용하면:

- **Planner**: 발산적 능력이 강한 모델. 다양한 전략적 대안을 탐색.
- **Executor**: 수렴적 능력이 강한 모델. 정밀한 코드 작성, 데이터 분석.
- **Critic**: 메타인지적 능력이 강한 모델. 결과물의 비판적 평가.

각 역할에 특화된 모델이 협업하는 구조. 인간 조직에서 기획자, 엔지니어, QA가 역할을 분담하듯이.

이것은 아직 초기 단계이지만, 2024-2025년에 이미 구체적인 형태를 갖추기 시작했다. Claude의 Computer Use, OpenAI의 Assistants API, 구글의 Gemini Agent — 각 진영이 에이전틱 아키텍처를 핵심 경쟁 영역으로 삼고 있다.

## 에이전트 시대의 조종사

에이전트의 등장은 조종사의 역할을 변화시킨다.

단발성 프롬프팅에서 조종사의 역할은 **"질문을 잘 하는 것"**이었다. 에이전트 시대에서 조종사의 역할은 **"시스템을 설계하는 것"**으로 격상된다.

- 어떤 도구를 에이전트에게 제공할 것인가?
- 계획-실행-비판 루프의 각 단계에 어떤 제약 조건을 부여할 것인가?
- 에이전트의 자율성 범위를 어디까지 허용할 것인가?
- 인간이 개입해야 하는 체크포인트를 어디에 설정할 것인가?

이것은 프롬프트 엔지니어링의 다음 단계이며, **시스템 아키텍처 설계**에 가깝다. 개별 질문의 밀도를 최적화하는 것(2-6절)에서, 전체 워크플로우의 구조를 최적화하는 것으로. 조종사의 역할이 "계기판 읽기"에서 "비행 계획 수립"으로 확장되는 것이다.

## 제3장을 마치며: 가면놀이의 교훈

다섯 개의 절에 걸쳐, 우리는 정체성의 논리를 해부했다.

- 페르소나가 잠재 공간의 특정 영역을 활성화하여 출력을 바꾼다(3-1).
- RLHF가 인간의 선호에 맞게 모델을 정렬하되, 과잉 정렬이라는 역설을 낳는다(3-2).
- CoT와 자기 수정이 메타인지의 기능적 등가물로서 추론 성능을 향상시킨다(3-3).
- 과제의 성격(발산/수렴)에 따라 최적의 모델과 전략이 달라진다(3-4).
- 에이전트가 단발성 추론을 넘어, 계획-실행-검증의 자율적 루프를 수행한다(3-5).

이 모든 것은 **"정체성"**의 논리였다. "나는 누구인가"라는 선언(페르소나)이 출력을 바꾸고, "나는 어떻게 행동해야 하는가"라는 정렬(RLHF)이 경계를 설정하며, "나의 사고가 올바른가"라는 반성(메타인지)이 품질을 끌어올린다.

그리고 이 모든 것의 최종 주체는 — 기차놀이에서도, 스무고개에서도, 가면놀이에서도 — **인간**이었다.

에필로그에서 그 인간에 대해 이야기하자.
